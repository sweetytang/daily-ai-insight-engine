# 项目归档

## 1. 题目理解

这道题考的不是“做一个摘要页面”，而是做一个最小可用的 AI 舆情流水线：

1. 获取近期 AI 新闻样本
2. 设计结构化 schema
3. 分步处理，而不是一次性把原始数据丢给模型
4. 生成日报分析
5. 给出可视化结果
6. 说明设计决策、AI 使用方式和开发方案

因此我把项目拆成了 `原始数据 -> 结构化抽取 -> 排序聚合 -> 日报生成 -> 可视化展示 -> 文档归档` 六段。  
在初版静态样例链路跑通之后，又补上了“实时抓取最新舆情”的能力，但仍然复用同一条分析链路，保证系统逻辑一致。
在此基础上，我又保留了原有规则版方案 A，并新增了真实调用大模型的方案 B，两者通过前端链接参数 `?solution=a|b` 切换，便于对比“规则链路”和“LLM 链路”的差异。

### 2.3 实时抓取来源

当前实时模式首版接入两个官方源：

- Anthropic News
  - 抓取主页新闻列表，提取标题、摘要、发布时间、链接
- Google DeepMind News
  - 抓取新闻列表页，提取标题、月份、链接，再补抓文章页 `meta description`

这样选择的原因：

- 这两个页面结构稳定，适合用简单规则解析
- 都是官方源，适合作为“最新舆情”输入
- 不依赖第三方 API key，部署和复现成本低

没有把 OpenAI 放进实时抓取首版，不是因为它不重要，而是因为当前服务端抓取会被 Cloudflare challenge 挡住。为了保证系统稳定，首版实时抓取先不硬接。

### 2.4 数据特点

- 官方源偏“事实公告”：适合提取事件类型、主题和关键实体
- 媒体源偏“市场情绪”：适合提取风险、竞争格局和情绪倾向
- 中英文混合：兼顾国际头部厂商和中国市场动态
- 实时源更偏“近期一手动态”：时效性强，但来源覆盖面比静态样例更窄
- 静态样例更偏“完整展示”：适合证明 schema、趋势分析和可视化能力

原始样例数据文件：`backend/services/data/raw-news.json`

## 3. Schema 设计

### 3.1 结构化字段

| 字段 | 作用 | 设计原因 |
| --- | --- | --- |
| `eventType` | 事件类型 | 支撑“模型发布 / 融资 / 合作 / 安全 / 基础设施”分类 |
| `primaryTheme` | 主主题 | 支撑聚合、趋势判断和可视化 |
| `secondaryThemes` | 次主题 | 保留复合事件，不把多模态/世界模型等信息丢掉 |
| `companies` | 涉及主体 | 支撑主体热度统计 |
| `keywords` | 关键词 | 支撑检索与人工校验 |
| `sentimentLabel` | 情绪倾向 | 区分正向、中性、需关注 |
| `impactScore` | 影响分 | 支撑热点 Top 排序 |
| `confidenceScore` | 置信度 | 区分官方公告和媒体转述的可信度差异 |
| `riskTags` | 风险标签 | 直接服务“风险提示”模块 |
| `opportunityTags` | 机会标签 | 直接服务“机会提示”模块 |
| `structuredSummary` | 结构化总结 | 用一句话归纳“这条新闻被归类为什么” |
| `impactAnalysis` | 影响分析 | 用一句话解释“为什么重要” |

### 3.2 为什么不能只做摘要

如果只有 `summary`，只能得到“每条新闻说了什么”，无法支撑下面这些题目要求：

- Top 3-5 热点排序
- 趋势判断
- 风险与机会提示
- 可视化聚合
- 过程可解释

所以这个 schema 的目标不是“写得更像文章”，而是“让日报可以计算”。

## 4. 关键步骤与理由

### 4.1 先清洗归一化

原因：

- 不同来源字段不统一
- 中英文混合会影响关键词识别
- 发布时间、来源类型必须先统一，后面才能排序

### 4.2 先把“数据获取”与“分析链路”分开

原因：

- 静态样例和实时抓取只是输入不同，不应该影响后续分析逻辑
- 把变化最大的部分限制在数据获取层，后续维护成本更低
- 实时抓取失败时，不会破坏已有的日报生成逻辑

### 4.3 再做结构化抽取

原因：

- 题目明确禁止“一次性丢给 AI”
- 先抽结构，后续才有排序、聚类和聚合基础
- 单条新闻单独处理，比整包输入更容易校验
- 同时保留规则版与 LLM 版，可以更清楚地说明什么是“可解释规则抽取”，什么是“真实模型抽取”

### 4.4 用影响分排序，而不是人工拍脑袋

影响分由四部分组成：

- 来源权重
- 主题权重
- 实体权重
- 时效性

原因：

- 热点排序必须可解释
- 避免“哪条写得更热闹就排前面”

### 4.5 实时抓取要去重和容错

原因：

- 不同来源可能会出现相同标题或相同链接
- 某一个源失败时，不应该让整次刷新失败
- 实时模式最重要的是“能返回可用结果”，而不是“所有源都完美成功”

当前去重策略是：`sourceUrl + normalizedTitle`

### 4.6 保留方案 A，再增加方案 B

原因：

- 方案 A 已经稳定可运行，不应该因为接入模型而被推翻
- 方案 B 可以让 `Prompt` 真正参与运行时，而不是只做展示
- 两套方案共用同一批输入数据，更适合横向比较
- 通过 URL 参数切换，前端和后端都不需要做复杂分支 UI

当前切换方式是：`?solution=a` 走规则版，`?solution=b` 走 LLM 版。

### 4.7 趋势从聚合结果推，不从直觉推

原因：

- 趋势判断必须有逻辑支撑
- 只有先聚合 `theme`、`company`、`sentiment`，趋势才不是空话

### 4.8 可视化只做最关键四类

- 主题分布
- 来源结构
- 情绪分布
- 重点实体

原因：

- MVP 时间有限
- 这四类图已经足够支撑题目展示
- 再多会稀释重点

## 5. 系统设计思路

### 5.1 整体架构

```text
静态模式：
raw-news.json
  -> LangGraph 清洗节点
  -> LangGraph 结构化抽取节点
  -> 排序聚合与日报生成
  -> Prisma 落库
  -> Express API
  -> React 看板展示

实时模式：
Anthropic News / Google DeepMind News
  -> 抓取与解析
  -> 去重与统一 raw news schema
  -> 方案选择层（A/B）
  -> LangGraph 清洗节点
  -> 结构化抽取节点（规则 or LLM）
  -> 排序聚合与日报生成
  -> Prisma 落库
  -> Express API
  -> React 看板展示
```

### 5.2 关键工程决策

- 前后端分离：便于清晰展示数据层、分析层、展示层
- SQLite：MVP 足够轻，便于本地复现
- Prisma：负责结构化存取，不把数据处理逻辑写死在 SQL 中
- LangGraph：把分析链路拆成可见步骤，满足题目“体现处理逻辑”的要求
- React 看板：把结构化结果和分析结论同时展示，避免只给 JSON
- 保留静态样例模式：防止系统完全依赖网络抓取
- 实时抓取采用“按钮触发”：符合当前需求，也最简单可维护
- 实时抓取只接稳定官方源：先保证能用，再考虑扩充来源
- 每次刷新覆盖当前快照：避免旧样本和新样本混在一起影响排序结果
- 保留方案 A：保证默认链路稳定、可解释
- 新增方案 B：让 `Prompt` 在运行时真实生效
- 前端通过 `solution` 参数透传到后端：避免做两套接口
- LLM 调用统一通过 `@langchain/openai`：保持与 LangChain / LangGraph 技术栈一致
- 结构化抽取和日报文案交给模型，影响分、排序、图表继续走规则：保证结果可解释、可比较

### 5.3 关于 Prisma 7 的取舍

当前环境里：

- `prisma generate` 正常
- `PrismaClient` 运行时要求显式传入 driver adapter
- `prisma db push` 的 schema engine 在当前环境异常

因此本项目采用的方案是：

1. 使用 `prisma.config.ts` 管理连接配置
2. 使用 `@prisma/adapter-better-sqlite3` 作为运行时 adapter
3. 使用初始化脚本创建 SQLite 表结构
4. 后续查询、写入仍然全部走 Prisma Client

这个取舍的原因很简单：

- 保留 Prisma 作为 ORM
- 避免把整个项目卡死在 CLI 的环境问题上
- 保证项目可运行比“形式上必须走 db push”更重要

## 6. AI 使用方式

### 6.1 使用位置

- `LangGraph` 负责编排处理流程
- `promptTemplates.js` 保存抽取 Prompt 和日报 Prompt
- 方案 A：规则抽取 + 规则日报生成，Prompt 只作为设计展示
- 方案 B：通过 `@langchain/openai` 调用在线模型，Prompt 在运行时真实参与“结构化抽取 + 日报生成”
- 实时抓取模式下，AI 不负责“爬网页”，而是负责处理统一后的结构化输入
- 前端通过 `?solution=a|b` 选择不同方案，后端在分析入口处分流

### 6.2 Prompt 设计思路

抽取 Prompt 的目标：

- 严格限制输出字段
- 禁止模型补充原文没有的信息
- 要求显式返回空数组和低置信度，而不是强行猜
- 让方案 B 的输出可以稳定落到现有 schema 上

日报 Prompt 的目标：

- 先聚合再分析
- 排序要引用 impactScore 和时效性
- 风险和机会要同时输出
- 禁止摘要拼接
- 只改“理解与表达”，不改“排序与统计”口径

Prompt 文件：`backend/models/promptTemplates.js`

### 6.3 错误处理

- 单条新闻先规范化，降低字段污染
- 通过 `confidenceScore` 标记可信度
- 实时抓取按来源隔离，单源失败不拖垮整次刷新
- 通过去重避免重复新闻影响排序
- 接口层统一 error middleware
- 用最小测试覆盖结构化流水线和抓取解析逻辑
- 方案 B 缺少模型配置时直接报错，不静默回退到方案 A
- 通过 `@langchain/openai` 做结构化输出时，优先走更严格的方法，失败后再做兼容回退
- 如果兼容网关返回的 JSON 结构不完全稳定，会先做字段规整，再进入最终 payload

## 7. 核心流程说明

### 7.1 从原始数据到结构化结果

1. 读取 `raw-news.json`，或者实时抓取官方新闻源
2. 标准化标题、摘要、发布时间和检索文本
3. 根据 `solution` 参数选择规则版或 LLM 版抽取节点
4. 识别事件类型、主主题、关键词、主体、情绪
5. 计算影响分
6. 生成结构化总结与影响分析

### 7.2 实时舆情刷新流程

1. 前端点击“实时生成最新舆情”
2. 当前页面把 `solution` 参数一起带上，请求后端 `POST /api/report/refresh?solution=a|b`
3. 后端并发抓取 Anthropic News 和 Google DeepMind News
4. 提取标题、摘要、链接、时间并统一成 raw news schema
5. 对抓取结果去重
6. 送入既有 LangGraph 骨架继续处理，但在抽取和报告节点按方案分流
7. 生成新日报并写入数据库
8. 前端拿到最新 payload 后刷新页面

### 7.3 从结构化结果到日报

1. 按影响分排序
2. 统计主题、来源、情绪、主体分布
3. 选出 Top 5 热点
4. 对 Top 3 做深度总结
5. 方案 A 用规则生成日报文案，方案 B 用大模型生成日报文案
6. 基于主题聚合生成趋势、风险和机会

### 7.4 从日报到可视化

1. 后端输出统一 payload
2. 前端读取 `/api/report/latest?solution=a|b`
3. 实时模式下也可以直接读取 `/api/report/refresh?solution=a|b` 的返回结果
4. 页面展示当前方案、实时抓取状态、Prompt 是否真实执行
5. 渲染概览卡片、热点、图表、结构化结果和方法说明

## 8. 开发方案

1. 先搭好 `frontend / backend / common / prisma / docs` 目录骨架
2. 补齐 React、Express、Prisma、LangGraph、Zustand、Sass 等依赖
3. 设计 Prisma 模型和原始新闻样本
4. 实现 LangGraph 处理链路
5. 实现 Express API 和日报持久化
6. 实现 React 看板和可视化
7. 生成样例日报
8. 在不推翻原链路的前提下补上实时抓取服务、刷新接口和前端按钮
9. 保留规则版方案 A，并新增基于 `@langchain/openai` 的方案 B
10. 用 URL 参数 `solution` 贯通前后端分流
11. 让 Prompt 在方案 B 中真实参与运行时调用，同时保留方案 A 的展示与对照价值
